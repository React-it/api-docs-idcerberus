# Organizacao do API Reference

O API Reference esta organizado por endpoint real da API. Alguns produtos usam
rotas proprias, enquanto a maioria das consultas externas passa pelo endpoint
central `POST /api/service-api`.

## Grupos principais

| Grupo | O que contem | Quando usar |
| --- | --- | --- |
| Autorizacao | `POST /api/token-generate` | Gerar JWT antes de chamadas protegidas |
| Integracao via SDK | TokenOnboarding, consulta de resultado e PDF | Integrar jornadas de onboarding |
| Servicos - Pessoas | Exemplos de `service` para CPF, biometria, risco e compliance | Executar consultas de pessoa fisica |
| Servicos - Empresas | Exemplos de `service` para CNPJ, socios e compliance | Executar consultas de pessoa juridica |
| Customers | Consulta e alteracao de status de clientes | Operar registros cadastrados |

## Por que varios produtos usam a mesma rota?

O endpoint `POST /api/service-api` funciona como uma entrada unica para varios
produtos. A rota nao muda; quem define o produto e o campo `service`.

Exemplo:

```json
{
  "service": "service_rfb_pf",
  "cpf": "cpf"
}
```

Para trocar de produto, mantenha a rota e altere o `service` e os campos
necessarios:

```json
{
  "service": "service_rfb_pj",
  "cnpj": "cnpj"
}
```

## Como navegar

1. Comece pelos guias quando ainda estiver entendendo o fluxo.
2. Use [Escolha o servico certo](/guides/escolha-o-servico-certo) para achar o service.
3. Use [Indice de services](/guides/indice-de-services) para copiar o nome exato.
4. Abra o API Reference para copiar request, response e schemas.
5. Use [Exemplos por ambiente](/guides/exemplos-por-ambiente) para conferir HML e producao.
