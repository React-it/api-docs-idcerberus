# Organização do API Reference

O API Reference está organizado por endpoint real da API. Alguns produtos usam
rotas próprias, enquanto a maioria das consultas externas passa pelo endpoint
central `POST /api/service-api`.

## Grupos principais

| Grupo | O que contém | Quando usar |
| --- | --- | --- |
| Autorização | `POST /api/token-generate` | Gerar JWT antes de chamadas protegidas |
| Integração via SDK | TokenOnboarding, consulta de resultado e PDF | Integrar jornadas de onboarding |
| Serviços - Pessoas | Exemplos de `service` para CPF, biometria, risco e compliance | Executar consultas de pessoa física |
| Serviços - Empresas | Exemplos de `service` para CNPJ, sócios e compliance | Executar consultas de pessoa jurídica |
| Customers | Consulta e alteração de status de clientes | Operar registros cadastrados |

## Por que vários produtos usam a mesma rota?

O endpoint `POST /api/service-api` funciona como uma entrada única para vários
produtos. A rota não muda; quem define o produto é o campo `service`.

Exemplo:

```json
{
  "service": "service_rfb_pf",
  "cpf": "cpf"
}
```

Para trocar de produto, mantenha a rota e altere o `service` e os campos
necessários:

```json
{
  "service": "service_rfb_pj",
  "cnpj": "cnpj"
}
```

## Como navegar

1. Comece pelos guias quando ainda estiver entendendo o fluxo.
2. Use [Escolha o serviço certo](/guides/escolha-o-servico-certo) para achar o service.
3. Use [Índice de services](/guides/indice-de-services) para copiar o nome exato.
4. Abra o API Reference para copiar request, response e schemas.
5. Use [Exemplos por ambiente](/guides/exemplos-por-ambiente) para conferir HML e produção.
